### Changed
- **Swapped the desktop GUI for a web server.** The frontend is now a browser UI served by an embedded HTTP server on `PORT` (default 8080). Rationale: simpler docker-compose deployment, no X11/Wayland in-container acrobatics. — 2026-04-22
  - `Cargo.toml`: removed `eframe`, `egui_extras`, `image`, `ureq`; added `tokio` (rt-multi-thread, macros, time, sync, signal), `axum` 0.7, `tower-http` 0.6 (fs), `reqwest` 0.12 (rustls).
  - `src/discord.rs`: converted to async. `Client::fetch_messages` and `Client::add_reaction` now take `&self` and return `impl Future`. All four existing parser / encoder tests still pass unchanged.
  - `src/main.rs`: rewritten as a `#[tokio::main]` axum server. Routes: `GET /` (single-page app), `GET /api/config`, `POST /api/fetch`, `POST /api/preview`, `POST /api/poller/toggle`, `GET /api/poller/status`, `GET /api/poller/log`, `GET /images/*` (static files via `ServeDir`). Graceful shutdown on SIGINT.
  - `src/main.rs`: poller ported from `std::thread` to `tokio::spawn` using the same `Arc<AtomicBool>` stop flag pattern and a `tokio::sync::Mutex<VecDeque>` for the shared log.
  - `src/index.html` (new): single-page UI built on htmx 2 (CDN-loaded). Dark theme, four-panel layout (messages / convert / image / handles) plus a bottom log tail polled every 3 s. Character counter turns red past 2,200 (Instagram's cap). Clicking a message in the list fills the textarea and auto-seeds the handle table with its mention IDs.
  - Security note: **the web UI has no authentication.** Bot token stays env-only — there's no UI field to read or modify it. Bind the listener to localhost or a trusted docker network.
  - Smoke-tested: `PORT=18080 cargo run` boots, `GET /` returns 10 KB HTML, `GET /api/config` returns the expected JSON, `GET /api/poller/status` returns the right HTML fragment. Real Discord API paths untested locally (no token). — 2026-04-22
- `Dockerfile`: two-stage build (rust:1.92-slim → debian:bookworm-slim). Dummy-src layer primes the dep cache so `COPY src` only rebuilds the final binary. Runs as uid 10001, strips the binary, exposes 8080. — 2026-04-22
- `docker-compose.yml`: publishes **9010:8080** per user spec. Bind-mounts `./images` read-only, uses a named `state` volume for `/app/state/state.json`. `.env` is optional (`required: false`). Smoke-tested end-to-end: `docker compose up -d` brings up the service, `curl localhost:9010/` returns 10 KB, `/api/config` returns the expected JSON. — 2026-04-22
- `.dockerignore`: excludes `/target`, `/images` (bind-mounted at runtime, not baked into the image), `.git`, `.env`, the `*.md` docs, and the compose files. — 2026-04-22
- `README.md`: project overview, quick-start (docker and cargo), full env-var table, post-image naming convention, auto-react semantics, security note on the unauth'd web UI, known caption-transform limits, and a file-tree cheat sheet. — 2026-04-22
- `.github/workflows/deploy.yml`: CI deploy. Triggers on push to `master`/`main` (plus `workflow_dispatch`), SSHes into the target via `appleboy/ssh-action@v1.2.0`, runs `git pull --ff-only` and `docker compose up -d --build`, then `docker image prune -f` to avoid layer bloat. `concurrency: deploy` serializes deploys. ### Fixed
- `Dockerfile`: pre-create `/app/state` with `chown app:app` before `USER app`. Without this, Docker mounts the named `state` volume as root-owned, and the non-root `app` user can't write `state.json`, surfacing as `Permission denied (os error 13)` in the log when auto-react starts. Verified end-to-end: fresh `docker compose up -d` → `/app/state` is `app:app`, `touch` succeeds as uid 10001. **Operational note:** existing volumes keep their old ownership because volume population only runs on first create. To apply this fix to an already-broken deployment, run `docker compose down -v && docker compose up -d --build` (wipes the volume — only partial state at most, since bootstrap was failing before this fix). — 2026-04-22

### Changed (auto-react hardening + unattended operation)
- `src/state.rs`: added `reactions_done_by_channel: HashMap<channel, HashMap<message, BTreeSet<emoji>>>` + `has_reacted` / `record_reaction` / `clear_reactions` helpers. Written after every successful reaction so a crash mid-batch resumes without re-hitting the API. Two new tests (14 total in state). `BTreeSet` over `HashSet` for stable JSON output. — 2026-04-22
- `src/main.rs`: reaction loop now skips emojis already in `reactions_done_by_channel`, saves state after each successful call, and `tokio::time::sleep(REACT_DELAY = 1s)` between every `add_reaction` to stay well under Discord's message-reaction rate-limit bucket. On full completion for a message, `clear_reactions` drops the in-flight entry so the persisted map stays bounded. — 2026-04-22
- `src/main.rs`: **poller auto-starts on launch** when `DISCORD_BOT_TOKEN` is set. Makes the app fire-and-forget under docker-compose — operator no longer has to open the UI and tick a box after every restart. Empty token still starts the server but skips the poller silently (no 401 log spam). New `GET /api/poller/running` endpoint returns `"1"`/`"0"` for the UI to keep the checkbox synced with actual poller state. — 2026-04-22
- `src/index.html`: message list auto-refreshes every 30 s (`hx-trigger="click, load delay:200ms, every 30s"`). The auto-react checkbox polls `/api/poller/running` on load and every 5 s so the UI reflects reality (including the auto-started state, or a crashed poller). Selected message is restored across auto-refreshes via a `data-selected-id` on the container — you don't lose your place if the list refreshes while editing. — 2026-04-22
- `CLAUDE.md`: documented per-emoji idempotence, the 1 s pacing, the new auto-start semantics, and the updated state-file schema. — 2026-04-22
- Smoke-tested: no-token server returns `running=0`; fake-token server auto-starts the poller (log shows `auto-react started` + an expected 401 from Discord); `index.html` response contains the auto-fetch trigger and `syncPollerCheckbox` function. 23/23 tests green. — 2026-04-22

### Added (gateway / bot-online presence)
- `src/gateway.rs` (new): minimal hand-rolled Discord Gateway v10 client (`tokio-tungstenite` + `futures-util`). Connects, IDENTIFYs with `intents: 0`, runs the heartbeat loop, reconnects on disconnect with exponential backoff (1 s → 60 s cap). Fatal close codes (4004 invalid token, 4010–4014 invalid shard/intents/API version) stop the loop so a misconfigured deployment doesn't spin forever. Holding this connection is what makes the bot appear **online** in Discord — REST alone doesn't affect presence. — 2026-04-22
- `Cargo.toml`: added `tokio-tungstenite` 0.24 (`connect` + `rustls-tls-native-roots`) and `futures-util` 0.3 (`std` + `sink`). — 2026-04-22
- `src/main.rs`: spawn `gateway::run` on startup alongside the poller when the token is set; new `AppCtx.gateway_connected: Arc<AtomicBool>` + `GET /api/gateway/status` endpoint returns an HTML badge (`online`/`offline`). — 2026-04-22
- `src/index.html`: new `bot:` topbar chip that polls `/api/gateway/status` every 5 s. — 2026-04-22
- Smoke-tested with a fake token: gateway connects to Discord, gets close-code 4004 on identify, logs `gateway: fatal, not reconnecting — closed: 4004 Authentication failed.` — exactly one attempt, no loop. With a real valid token the bot shows online in Discord. — 2026-04-22

### Changed (gateway-triggered real-time reactions)
- `src/gateway.rs`: IDENTIFY now requests `GUILD_MESSAGES` (`IDENTIFY_INTENTS = 1 << 9`, non-privileged) instead of `intents: 0`. The receive loop now handles `MESSAGE_CREATE`: on a match for `ctx.channel_id`, pushes a log line and calls `ctx.poll_trigger.notify_one()`. `MESSAGE_CONTENT` is deliberately NOT requested — it's privileged and unnecessary since we fetch bodies via REST. — 2026-04-22
- `src/main.rs`: new `AppCtx.poll_trigger: Arc<tokio::sync::Notify>`, initialized in `main()`, passed to both the gateway spawn and the poller. The poller's inter-fetch sleep now uses `tokio::select!` so the 30 s timer races the trigger — reactions happen within a few seconds of a new announcement instead of waiting out the full cycle. The timer remains the safety net if the gateway briefly drops. — 2026-04-22
- `CLAUDE.md`: documented the intent change and the trigger semantics. — 2026-04-22
- Smoke-tested: build clean, 23/23 tests green, fake-token run still reports `gateway: fatal, not reconnecting — closed: 4004` exactly once. — 2026-04-22
